🧠 What Is Nexus AI?

Nexus AI is not a single finished website. It is a headless, serverless, multi-surface site-building framework — the operational engine that generates, deploys, and governs any site, for any client or product, from one central control plane.

The framework wires five layers into a single orchestrated pipeline:

User describes a site in plain language
           │
           ▼
┌──────────────────────┐
│   STITCH (Design)    │  Generates screens, DESIGN.md, React/Tailwind
│   stitch.withgoogle  │  exports via MCP into the repo
└──────────┬───────────┘
           │ MCP export → DESIGN.md committed to repo root
           ▼
┌──────────────────────┐
│  ANTIGRAVITY (Code)  │  Reads DESIGN.md + repo context, plans,
│  Agent-first IDE     │  executes, tests, and iterates autonomously
└──────────┬───────────┘
           │  — or parallel fast-path ↓
           ▼
┌──────────────────────┐
│  LOVABLE (Scaffold)  │  Full-stack generation from prompt — frontend,
│  lovable.dev         │  backend, auth, DB — syncs to GitHub on create
└──────────┬───────────┘
           │ All code paths converge here ↓
           ▼
┌──────────────────────┐
│  GITHUB (Source)     │  Repo creation, branch, CI/CD pipeline,
│  github.com          │  PR automation, webhook events back to Nexus
└──────────┬───────────┘
           │ GitHub Actions quality gates pass ↓
           ▼
┌──────────────────────┐
│  FIREBASE (Runtime)  │  Firestore, Auth, App Hosting, Cloud Run —
│  for generated sites │  auto-provisioned per site by Firebase adapter
└──────────────────────┘
           │
           ▼
   Live URL delivered to user
   Nexus admin dashboard shows status

Nexus does not prescribe a finished design for itself. The visual output is entirely determined by the client prompt flowing through Stitch. Nexus is the engine, the governance layer, and the operational contract.

Guiding Principles
Headless by default. Route shells, UI, workflows, and AI calls are decoupled through typed contracts.
Serverless by default. Stateless request handlers and background jobs unless a long-running service is clearly justified.
Surface-aware experience. Layout, motion, focus, and input assumptions change by capability profile.
Progressive enhancement. Modern APIs improve the experience; the baseline remains functional without them.
Fewer services, fewer hops. No n8n, no external orchestration. Postgres-backed internal workflow queue only.
Security before convenience. Multi-tenant isolation, auditability, and deletion workflows are first-class.
Performance is a feature. Budgets enforced in CI, monitored in production.
Graceful degradation over brittle novelty. All advanced motion, voice, and AI must fail safe.
Explicit Non-Goals (v5.1)
Native iOS or Android applications
Dedicated tvOS or console apps
Full offline-first application behaviour
External workflow SaaS (n8n or equivalent)
Billing engine
End-user AI analytics dashboard
🗄️ Data Layer Map

The most important architectural clarification in this codebase. Two databases. Two completely separate concerns. They do not compete.

┌─────────────────────────────────────────────────────────────────────┐
│          LAYER 1 — NEXUS CONTROL PLANE (PostgreSQL 16)              │
│                    Managed by Payload CMS                           │
│                                                                     │
│  What lives here:                                                   │
│  • Payload collections (prompts, tenants, users, workflowJobs,     │
│    auditLogs, pages)                                                │
│  • pgvector HNSW embeddings for RAG retrieval                       │
│  • Internal async job queue (workflow_jobs table)                   │
│  • AI governance audit trail                                        │
│  • Nexus platform users and tenant configurations                   │
│  • Feature flags, site settings, assistant defaults                 │
│                                                                     │
│  Multi-tenant isolation: Row Level Security on all tenant tables    │
│  Access pattern: Payload Local API (server-side only)               │
│  Location: Self-hosted or managed PostgreSQL 16                     │
└─────────────────────────────────────────────────────────────────────┘


┌─────────────────────────────────────────────────────────────────────┐
│       LAYER 2 — GENERATED SITE RUNTIME (Firebase Services)          │
│              Provisioned per-site by Firebase adapter               │
│                                                                     │
│  What lives here:                                                   │
│  • Firestore — real-time database for each generated site           │
│  • Firebase Auth — Google Sign-In, email/password for site users    │
│  • Firebase App Hosting / Cloud Run — live site deployment          │
│  • Cloud Secret Manager — per-site credential storage               │
│  • Firebase Hosting — CDN for static assets                         │
│                                                                     │
│  Each generated site gets its own Firebase project.                 │
│  Firebase adapter auto-provisions services based on site config.    │
│  This layer has no role in Nexus platform internals.                │
└─────────────────────────────────────────────────────────────────────┘

Rule: Never use Firestore for Nexus platform data. Never use PostgreSQL as the runtime database for generated sites. The boundary is hard and intentional.

🌐 Multi-Surface Support

Surface detection runs in three mandatory stages. Edge classification is never final. Client arbitration always runs after hydration.

Stage 1: Edge guess        UA + Client Hints → conservative initial classification
Stage 2: Client arbitration matchMedia, pointer, viewport, touch points, feature detect
Stage 3: User override     Manual selection persisted in localStorage / cookie
Surface	Devices	Input	Motion	Min Font
Smartphone	phones, compact foldables	tap, swipe, optional voice	reduced	16px
Tablet	iPad, large tablets	touch, stylus, keyboard	moderate	15px
Desktop	laptops, workstations	mouse, keyboard, optional voice	full	14px
Cinematic	Smart TVs — Tizen 7+, webOS 23+, Android TV	D-pad remote, limited voice	minimal	24px

TV-specific requirements enforced in code:

Every focusable reachable by arrow keys — no dead ends, no traps
Focus ring always visible — never disappears off-screen
Overscan-safe margins on all Cinematic routes
Parallax, WebGL, and heavy animation disabled by default on Cinematic surface
TV traversal tested in Playwright AND in Samsung Tizen / LG webOS emulators before release
Final sign-off requires at least one real-device validation pass
🔗 Integrations

All five adapters follow the same contract: typed interface, health check, graceful degradation. No adapter is a hard runtime dependency. All adapters are server-side only. Credentials never reach client bundles.

Central registration: src/lib/integrations/registry.ts

INT-1 · GitHub

Role: Source control, CI/CD trigger, repo management, PR automation, webhook receiver — for the Nexus repo and every site it generates.

Capability	Detail
Repo management	Create, clone, fork, list repos for a tenant or org
Branch operations	Create branches, push commits, open and merge PRs
Actions dispatch	Trigger workflows, poll run status, retrieve logs
Deployment events	Read deployment status and commit statuses
Webhooks	Receive push/PR/deployment events → enqueue to workflow_jobs
Signature verification	HMAC on every incoming payload — never skipped

Auth: GitHub App preferred over PATs — granular permissions, longer token lifetime.

🚀 Quick Start
Prerequisites
Node.js 20+
npm 10+
PostgreSQL 16 with pgvector extension
Git
A Google account — Stitch and Antigravity are free with a Google account
Step 1 — Clone and install
git clone https://github.com/YuryZZZ/nexus-ai-platform.git
cd nexus-ai-platform
npm install
Step 2 — Configure environment
cp .env.example .env.local

Open .env.local. The minimum required to boot:

DATABASE_URL=postgresql://user:password@localhost:5432/nexus_ai
PAYLOAD_SECRET=your-payload-secret-minimum-32-characters
NEXTAUTH_SECRET=your-nextauth-secret
NEXT_PUBLIC_SERVER_URL=http://localhost:3001
GLM_API_KEY=your-glm-api-key

To unlock the full integration pipeline also add:

GITHUB_TOKEN=
STITCH_API_KEY=
ANTIGRAVITY_API_KEY=
GOOGLE_AI_STUDIO_API_KEY=
FIREBASE_SERVICE_ACCOUNT_KEY=
FIREBASE_PROJECT_ID=
LOVABLE_API_KEY=
LOVABLE_ORG_ID=
Step 3 — Set up the database
# Create the database
createdb nexus_ai


# Enable pgvector
psql nexus_ai -c "CREATE EXTENSION IF NOT EXISTS vector;"


# Run Payload migrations (auto-runs on first boot — manual trigger for CI)
npm run db:migrate


# Apply Row Level Security policies to all tenant-scoped tables
npm run db:rls


# Seed initial data — admin user, default tenant, feature flags, starter prompts
npm run db:seed
Step 4 — Run the development server
npm run dev -- -p 3001
URL	Purpose
http://localhost:3001	Marketing site
http://localhost:3001/app	Authenticated application
http://localhost:3001/admin	Payload CMS admin panel
http://localhost:3001/api/integrations/health	Integration health (admin session required)
Step 5 — Verify integrations
# Check all five adapter health checks
curl http://localhost:3001/api/integrations/health \
  -H "Cookie: <your-admin-session-cookie>"

Expected response:

{
  "timestamp": "2026-03-26T00:00:00.000Z",

    "github":      { "status": "healthy", "latencyMs": 142 },

🏛️ Payload CMS

Payload 3 is the Nexus control plane — unified content management, AI governance, and operational data. It does not replace PostgreSQL; it runs on top of it via the Postgres adapter.

Collection Reference
Collection	Group	Purpose
prompts	AI Governance	Versioned AI prompt registry with approval workflow and risk levels
tenants	System	Tenant metadata, motion intensity defaults, voice policy
users	System	Auth-enabled users — tenant-aware, role-based (admin / editor / viewer)
workflowJobs	Operations	Async job visibility and status tracking
auditLogs	Operations	AI decisions, access events, GDPR deletion audit
pages	Content	CMS-managed pages with Lexical rich-text editor
Globals Reference

Local API Safety Rule

The Payload Local API skips access control by default. Always pass overrideAccess: false and user for any call that must respect permissions:

const result = await payload.find({
  collection: 'prompts',
  where: { slug: { equals: 'default-assistant' } },
  overrideAccess: false,   // ← required for access-controlled reads
  user,
  depth: 0,
})
Payload + RLS Boundary

Payload access control handles application-layer permissions. Postgres RLS handles database-layer tenant isolation. Both are required. Payload ACL does not replace RLS. RLS must be enabled and tested on all tenant-scoped tables regardless of what Payload enforces above it.
